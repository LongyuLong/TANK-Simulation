# 채점기(Scoring Function) 설계 — 인수인계

> 목적: 전차 글로벌 경로의 "좋음"을 **객관적으로 채점**하는 함수를 설계한다.
> 이 채점기를 목적함수로 삼아 A* 비용맵 가중치를 최적화한다(Option A: 목적함수 + 베이지안/CMA-ES).
> 교리 근거: 미 육군 **ATP 3-20.15 Tank Platoon (2025)**.

---

## 0. 왜 이걸 하는가 (맥락)

- 현재 A* 글로벌 플래너는 휴리스틱 비용맵(`cost_risk_v2`)으로 경로를 낸다.
- 문제: 출발(5,5)→목적(55,55) 리스크 경로가 **언덕을 그냥 넘어버린다**. 우회 비용보다 횡단 비용이 싸게 산정됐기 때문.
- 가중치를 손으로 바꾸면 경향은 바뀌지만 **일회성**이라 시스템이 아니다.
- 그래서: "좋은 경로"를 채점하는 **목적함수**를 먼저 만들고, 그걸로 가중치를 자동 튜닝한다.

### ⚠️ 가장 중요한 설계 원칙 (순환논리 금지)
채점기(목적함수)는 비용맵(`ridge/open/los_risk`)과 **같은 레벨이면 안 된다**.
- **비용맵** = "이 셀이 위험할 것 같다"는 *추정/대리지표(surrogate)*
- **채점기** = "실제로 탐지·피격됐는가"라는 *결과(ground truth)*
- 채점기는 비용맵보다 **한 단계 더 근본적인 물리량**(탐지확률·피격확률·기동시간)이어야 한다.
- 그래야 "비용맵 가중치를 결과에 맞춰 보정한다"는 작업이 의미를 가진다.

## 1. 현재 시스템 상태 (재사용 가능한 자산)

**패키지**: `tanknav/` (config, mapio, viz, mapping, risk, planning, perception)
**격자**: 61×61, `GRID_RES=5m`, 맵 300m×300m

**보유 데이터 레이어** (모두 `.npy`, `fin/` 루트):
| 레이어 | 의미 |
|---|---|
| `heightmap` (m) | 고도 |
| `slope` (°) | 경사도 |
| `cost` (base v3) | 경사+절벽+장애물+inflation, inf=통과불가 |
| `obstacle` (bool) | 장애물 |
| `ridge_risk` (0~1) | 능선 노출(피탐지) |
| `open_risk` (0~1) | 개활지 노출(엄폐 부족) |
| `los_risk` (0~1) | 적 가시선 직접 노출 |

**적 모델** (`risk.Enemy`): infantry(탐지 500m, 관측높이 1.8m) / tank(2000m, 2.4m) / patrol(경로).
**LoS 계산**: `risk.is_visible()` — Bresenham + 지형 차폐(고도 보간) 이미 구현됨.
**K2 제원**: 전고 2.4m, 전폭 3.6m.

## 2. 교리 근거 (ATP 3-20.15에서 추출)

### 2-1. OAKOC — 지형의 5대 군사적 요소 (¶2-48) ★채점기의 뼈대
| 요소 | 교리 정의 | 채점기 매핑 |
|---|---|---|
| **O**bservation & fields of fire (¶2-63~64) | 피아 관측/사계, intervisibility line | **탐지확률** (LoS 노출) |
| **A**venues of approach (¶2-57) | 기동로, 통행성으로 분류 | **기동시간** (속도) |
| **K**ey terrain (¶2-60) | 점령 시 이점 | 임무 보너스(선택) |
| **O**bstacles (¶2-52) | 자연(강·산·삼림·연약지반)/인공 | **통과불가(inf)** |
| **C**over & **C**oncealment (¶2-67) | **Cover=화력 방호 / Concealment=관측 차폐** | **피격확률↓ / 탐지확률↓** (★별개!) |

> **핵심 통찰 (¶2-67)**: Cover ≠ Concealment.
> - **Concealment**(은폐) = 관측으로부터 보호 → **탐지확률**을 낮춤
> - **Cover**(엄폐) = 화력으로부터 보호 → 탐지돼도 **피격/살상확률**을 낮춤
> → 채점기에서 두 효과를 **분리된 항**으로 모델링해야 한다.

### 2-2. 통행성 3단계 (¶2-54~56) → 기동시간/이동비용
- **Unrestricted**: 평지~완경사, 장애물 희박 → 자유기동(저비용/고속)
- **Restricted**: 중~급경사, 중밀도 장애물 → 지그재그·우회, 속도저하(중비용)
- **Severely restricted**: 급경사, 고밀도 장애물, 도로없음 → 종대 강요, 매우 느림(고비용/불가)

### 2-3. 최대교전선 & 2/3 규칙 (¶4-156) → 피격확률 거리 모델
- 무기 최대유효사거리 기준이되, **2/3 규칙** 적용 시 명중/살상확률(Ph/Pk)↑.
- 즉 **탐지거리 ≠ 효과적 살상거리**. 적 LoS 안이라도 거리에 따라 Pk가 다르다.

### 2-4. 사각(Dead space) (¶4-154) → 안전 회랑
- "직사화기로 관측·사격 불가한 구역" → 지형차폐로 **저위험 통로**. 경로가 선호해야 함.

### 2-5. 차폐진지(Defilade)/역사면 (¶4-146, fig 4-3)
- hull/turret defilade, hide position, 역사면 = 차체/포탑이 지형에 가려지는 지형.
- 노출 감소 지형 특징 → concealment/cover 모두 기여.

### 2-6. 기동 기법 = 속도↔보안 trade-off (Ch.3/Ch.5)
- traveling / traveling overwatch / bounding overwatch = 위협수준별 기동.
- 이게 곧 사용자가 말한 **생존성↔속도 trade-off**의 교리적 근거.

## 3. 채점기 설계 방향 (제안)

### 3-1. 목적함수 구조 — 탐지→피격 체인 (ground truth 레벨)
경로를 셀(또는 세그먼트) 단위로 보고:

```
P(생존) = Π_세그먼트 [ 1 − P(피격_세그먼트) ]

P(피격) = P(탐지) × P(교전|탐지) × P(명중|교전) × P(살상|명중)
```
각 항의 입력(보유 레이어로 계산 가능):
- **P(탐지)**: 적 LoS(is_visible) × 거리감쇠 × **은폐(concealment)** × 노출시간(셀 체류) × [열교차·날씨]
- **P(교전|탐지)**: 최대교전선/2/3 규칙 — 거리 기반
- **P(명중|교전)**: 거리 기반 Ph
- **P(살상|명중)**: **엄폐(cover)/차폐진지**로 감소

### 3-2. 기동시간 목적
```
T = Σ 세그먼트길이 / 속도(통행성 tier)
```

### 3-3. 다목적 결합 (임무 의존)
```
Score = w_surv · (1 − P생존) + w_time · T_정규화
```
- 공격: 속도·저시계 기동 가중 ↑
- 방어/정찰: 생존성 가중 ↑
→ **이 trade-off 가중치 결정이 이번 설계의 핵심 정책 결정.**

### 3-4. "언덕 넘기" 문제가 자연히 풀리는 이유
능선 횡단 셀은 적 관측/silhouette로 **P(탐지)가 급등** → 생존성 목적함수가 자동으로 페널티.
우회가 시간을 약간만 늘리고 생존을 크게 높이면, 최적화가 `W_RIDGE`/관련 가중치를 올린다.
→ 손튜닝이 아니라 **시스템**으로 해결.

## 4. 이 채팅에서 결정/산출할 것

1. **목적함수 확정**: 3-1 체인의 각 P(·)를 보유 레이어로 어떻게 계산할지 구체식.
   - 특히 concealment vs cover를 어떤 데이터로 구분할지 (현재 비전 지형분류 미구현 → 우선 고도/경사/장애물 근사).
2. **거리-확률 곡선**: 최대교전선/2/3 규칙을 P(교전)·Ph·Pk 곡선으로 수식화.
3. **노출시간 모델**: 셀 체류시간 = 셀길이/속도(tier). 정지·저속 구간 가중.
4. **생존성↔속도 가중치**(`w_surv`,`w_time`) 정책: 임무유형별 프리셋.
5. **순환논리 회피 검증**: 목적함수가 비용맵 항을 그대로 쓰지 않는지 점검.
6. **산출물**: `tanknav/eval.py` 설계 — `score_path(path, enemies, mission) -> dict(P_survive, time, score)`.
   - 이후 `tanknav/tune.py`에서 CMA-ES/베이지안 최적화로 비용맵 가중치 탐색.
